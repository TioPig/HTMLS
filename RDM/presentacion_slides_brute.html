Equipo, Problema y Objetivo

¿Quiénes somos y qué necesidad estamos resolviendo?

Bloque 1 · Contexto (3 min)
Equipo de desarrollo

Desarrollador 1 Fullstack & Líder Técnico

Desarrollador 2 Fullstack & QA / DevOps

Responsables de frontend, backend, base de datos, despliegue y documentación técnica.

Problema
  • Recetas familiares y ancestrales dispersas, sin un repositorio confiable ni estandarizado.
  • Restaurantes y chefs sin una herramienta robusta para gestionar y compartir recetas.
  • Falta de valoración, comentarios estructurados y métricas de popularidad.
Objetivo de Recetas Del Mundo

Rescatar, organizar y potenciar el patrimonio culinario mundial conectando recetas de madres, abuelas, cocineros y chefs con restaurantes y comensales.

Lo hacemos mediante un sistema web profesional que permite publicar, valorar, comentar, marcar como favorito y gestionar recetas de manera segura y escalable.

Valor que entregamos
  • Plataforma centralizada para recetas con datos completos y multimedia.
  • Funciones sociales: valoraciones, favoritos, comentarios y donaciones.
  • Arquitectura lista para crecer y ser utilizada en entornos reales.

Arquitectura del Sistema

Cómo se comunican los componentes principales.

Arquitectura multicapa

En la presentación original puedes ampliar esta imagen si lo necesitas.

Flujo simplificado
  • El usuario interactúa con la SPA en React (frontend).
  • React consume la API REST de Spring Boot (controladores REST).
  • La lógica de negocio se implementa en servicios y repositorios JPA.
  • Los datos se persisten en PostgreSQL mediante entidades y relaciones normalizadas.
  • Spring Security + JWT protege los endpoints sensibles.
Ventajas de esta arquitectura
  • Separación clara entre presentación, negocio y persistencia.
  • Escalabilidad: frontend y backend pueden escalarse de forma independiente.
  • Fácil mantenimiento y extensión de funcionalidades.

Modelo de Datos

Cómo se estructuran usuarios, recetas y valoraciones.

13 tablas · FKs validadas

Tablas principales: usuarios, recetas, países, categorías, valoraciones, etc.

Relaciones clave
  • Un usuario puede crear muchas recetas y muchas valoraciones.
  • Cada receta está asociada a un país y una categoría.
  • Las valoraciones vinculan usuarios con recetas (estrellas, comentarios).
Características del modelo
  • Modelo normalizado que evita duplicidad de datos.
  • Uso de claves foráneas y restricciones de integridad.
  • Índices en campos de búsqueda (nombre, país, categoría).

API REST & Autenticación

Resumen de endpoints y seguridad JWT.

90+ endpoints
Grupos de endpoints
/api/auth · login / registro /api/recetas · CRUD recetas /api/paises · países /api/categorias · categorías /api/valoraciones · estrellas y comentarios /api/favoritos · favoritos por usuario

Todos los endpoints están documentados con Swagger / OpenAPI.

En la defensa puedes mostrar solo 1–2 ejemplos en Swagger para no gastar tiempo.

Seguridad con JWT

Política: todos los POST / PUT / DELETE requieren un token JWT válido.

Header: Authorization: Bearer <token>

Roles: USER, ADMIN, LIDER · cada uno con permisos diferentes sobre los recursos.

  • Contraseñas encriptadas con BCrypt.
  • Configuración de CORS para permitir solo orígenes confiables.
  • Manejo de errores y respuestas claras para 401 / 403.

Despliegue con Docker

Cómo ejecutamos todo el sistema de forma integrada.

Entorno listo para producción
Estrategia de containerización
  • Contenedor backend (Spring Boot) con imagen ligera usando multi-stage build.
  • Contenedor frontend servido mediante Nginx.
  • Contenedor PostgreSQL con volúmenes persistentes.
  • Proxy reverso configurado para enrutar tráfico al frontend/backend.

El sistema puede levantarse de forma automática con un solo comando de Docker Compose.

Beneficios del despliegue
  • Reproducible: mismo entorno en desarrollo y producción.
  • Escalable: posibilidad de escalar servicios por separado.
  • Mantenible: actualizaciones controladas por versión de imagen.

Testing y Calidad

Cómo validamos que los flujos críticos funcionan.

Pruebas unitarias · integración · E2E
Tipos de pruebas realizadas
  • Unitarias: servicios de negocio en Spring Boot (validación de reglas y cálculos).
  • Integración: endpoints REST contra la base de datos (flujos de recetas, usuarios, valoraciones).
  • E2E: scripts completos probando login, búsqueda de recetas, valoración y favoritos.

Los flujos críticos de la aplicación fueron validados: autenticación, creación/edición de recetas y operaciones de valoración.

Esto reduce el riesgo de errores en producción y facilita futuras mejoras.

Calidad de experiencia de usuario
  • Revisión de UX: consistencia de colores, tipografías y estados visuales.
  • Pruebas en distintos tamaños de pantalla (móvil, tablet, desktop).
  • Manejo de errores y mensajes claros al usuario.

Metodología Ágil y Timeline

Cómo organizamos el trabajo a lo largo del proyecto.

Scrum · 3 sprints
Metodología de trabajo
  • Uso de Scrum adaptado: backlog, planning, daily, review y retrospectiva.
  • 3 sprints de 4 semanas (total 12 semanas).
  • Comunicación constante y revisión de prioridades según avance.

El foco estuvo en entregar valor funcional al final de cada sprint (MVP incremental).

Timeline simplificado
  • Fase 1: Investigación y levantamiento de requerimientos.
  • Fase 2: Diseño de UX/UI, arquitectura y modelo de datos.
  • Fase 3: Implementación base de datos y backend.
  • Fase 4: Desarrollo frontend y conexión con API.
  • Fase 5: Pruebas, ajustes, despliegue en Docker y documentación.

Conclusiones y Trabajo Futuro
